# Edge Scanner 文档工作流

项目名称：Edge Scanner

## 1. 项目启动指令

后续如需继续让 AI 扩展本项目，请先发送以下指令：

```text
请严格按照以下流程执行：

1. 先写 PRD
2. 再写架构文档
3. 再写接口文档
4. 再写实现规范
5. 再写测试用例
6. 再写代码
7. 再写测试
8. 最后写项目总结
全程必须参考前面已确认的文档，不允许自由发挥。
```

## 2. 执行约束

- 文档顺序固定：`PRD -> ARCH -> API -> SPEC -> TEST CASE -> CODE -> TEST -> SUMMARY`
- 后置文档必须引用前置文档，不允许新增未定义字段、接口、流程或模块。
- 当前仓库代码已存在，本轮产出以“反向归档现有实现”为准，不对业务逻辑作额外假设。
- 代码和测试扩展时，必须优先更新本目录下文档，再实施变更。
- 进入任何测试、回归、验收或 Provider 校验任务前，必须先读取 `10_TEST_PROGRESS.md`，优先执行其中仍为“待执行 / 部分验证 / 阻塞”的项目，并在本轮结束前回写状态、命令和结果。
- 自定义 Provider 的接口校验属于 `CODE` 和 `TEST` 阶段的强制动作，不能只依赖离线单元测试。

## 2.1 自定义 Provider 校验规则

- 对每个自定义 Provider，至少执行一次“定向测试 + Provider-only 扫描”。
- 如果发现返回为空、字段异常、盘口映射不对、赔率明显异常或状态码异常，必须先查看官方文档，再决定是否修改代码。
- 官方文档必须作为 Provider 接口字段、端点、分页和鉴权方式的主依据；第三方博客、论坛和旧快照只能作为辅助参考。
- 最终项目总结前，必须抽查扫描结果中的顶部套利 / 中间盘 / +EV 结果，确认：
  - 赛事双方匹配正确
  - 盘口类型匹配正确
  - 赔率来源和事件链接合理
  - 异常高 ROI 是否只是小额流动性或过期盘
